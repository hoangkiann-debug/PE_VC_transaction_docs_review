# FAQ And Troubleshooting

本文件把常见障碍统一转换为用户能直接执行的下一步。对用户说明问题时使用下列编号，但不要只输出技术错误或堆栈信息。

## 标准说明格式

每个问题都按四项说明：

1. **编号与问题**：发生了什么。
2. **影响**：哪些文件或判断尚未完成。
3. **下一步**：用户现在可以提供或操作什么。
4. **可继续部分**：当前仍能可靠完成什么。

## 应避免的错误用法

| 不建议这样做 | 会造成什么问题 | 应当怎样做 |
|---|---|---|
| 没有文件或条款原文，却要求给出确定修改意见 | 无法定位条款、定义和版本 | 先提供文件或可定位原文；只有背景讨论时明确标为一般性分析 |
| 用一份当前稿判断对方是否接受了上一轮意见 | 缺少版本和既有立场基线 | 补充上一版、问题清单或对方回复；否则只做当前文本风险审阅 |
| 把OCR失败、加密或损坏文件当成已审阅 | 会形成虚假的完整性结论 | 标记未实质审阅，主动列出替代材料并继续其他文件 |
| 把市场采用比例当成法律结论 | 市场惯例不等于合法或可执行 | 市场数据只作谈判背景，法律问题另行核验一手依据 |
| 同时加载全部参考文件和大型JSON | 增加干扰并降低条款定位质量 | 按三层导航读取；大型JSON只用查询脚本 |
| 未经确认把单个项目让步写成长期偏好 | 可能把特殊妥协错误应用到其他项目 | 只有用户明确确认后才写入审阅偏好表 |
| 直接覆盖用户原始Word文件 | 破坏审计链并增加不可逆风险 | 另存批注稿或红线稿，并保留源文件校验信息 |

## 问题与修复路径

| 编号 | 问题 | 对用户的下一步 | 当前可继续部分 |
|---|---|---|---|
| INPUT-001 | 没有收到文件或可定位的条款文本 | 上传文件、文件夹，或粘贴需审阅条款 | 仅可确认审阅范围和所需材料 |
| SCOPE-001 | 无法判断代表哪一方 | 告知代表公司/创始人方、投资人或其他立场 | 可先建立文件地图，不作立场性修改建议 |
| READ-001 | 文件损坏、格式不支持或提取为空 | 优先提供可打开的Word、可搜索PDF或纯文本版本 | 继续审阅其他可读文件，并列明未审文件 |
| READ-002 | 文件加密或受密码保护 | 由用户在本地解密后重新提供；不要要求用户发送密码 | 继续审阅未加密文件 |
| OCR-001 | 扫描PDF无法完成OCR | 提供带文字层PDF、原始Word或清晰分拆扫描件 | 该文件标记为未实质审阅，其他文件继续 |
| VERSION-001 | 后续红线稿缺少上一版或上一轮立场 | 补充上一版、问题清单、批注或谈判回复 | 可做当前文本风险审阅，但不判断接受/拒绝状态 |
| PACKAGE-001 | 缺少章程、side letter、披露函或其他关键配套文件 | 补充缺失文件，或确认接受有限审阅 | 对现有文件审阅并明确无法完成的跨文件检查 |
| AUTHORITY-001 | 法律依据过期、状态不明或无法核验 | 允许补充检索，或由承办律师核验最新一手依据 | 可提供问题识别和谈判建议，不作确定法律结论 |
| COMMENT-001 | Word批注锚点不唯一或批注写入失败 | 提供可编辑Word，或接受批注计划表 | 输出完整批注计划和可直接粘贴的建议文字 |
| CONNECTOR-001 | 外部数据库或连接器不可用 | 提供官方材料或稍后重试；不得要求提供账号密码 | 使用当前文件及本地资料完成可验证部分 |
| OUTPUT-001 | 输出校验未通过 | 修复缺失定位、建议条款、Fallback或文件冲突后重新校验 | 不把未通过校验的版本标记为最终交付 |

## OCR 专项修复顺序

1. 先确认PDF能否正常打开、是否加密、是否已有可搜索文字层。
2. 无文字层时优先运行 `scripts/ocr_pdf.py` 自动选择可用引擎；macOS可使用Vision，Windows/Linux/macOS也可使用本机OCRmyPDF/Tesseract。工具会返回 `OCR-*` 编号及下一步。
3. OCR只识别部分页面时，列出失败页码；除非用户明确接受，不把部分结果当成全文审阅。
4. OCR仍失败时，请用户提供原始Word、带文字层PDF或更清晰扫描件。
5. 交付中列明该文件未实质审阅及其对跨文件判断的影响。
6. 不等待用户追问：发生 `OCR-001` 时，主动列出“带文字层PDF、原始Word、清晰分拆扫描件”三种替代材料，并继续处理其他可读文件。
7. `scripts/ocr_pdf.py` 的自动模式应依次尝试全部可用本地引擎，而不是第一个引擎失败就停止。全部失败时输出 `OCR-MANUAL-001`，将上述三种替代材料作为面向用户的接续步骤。

## 外部连接重试与超时

- 同一只读查询最多自动重试2次，重试前确认失败不是权限、参数或登录问题。
- 单次查询超过30秒仍无结果时停止等待，标记 `CONNECTOR-001`，继续本地可完成部分。
- 权限、登录、验证码、付费或上传限制不自动重试，也不要求用户提供密码或验证码。
- 每次重试使用相同的最小查询范围，避免重复提交或扩大数据传输。

## 本地运行与异常边界

- 核心审阅默认使用本地文件和Skill内置数据，不以任何境外API或外部数据库可用为前提。
- `scripts/runtime_self_check.py --format json` 在处理用户文件前检查Python版本、核心数据及可选组件；该检查不联网，也不读取用户交易文件。
- 文档提取、版本比较、市场查询、法律依据查询和输出校验均在命令行边界捕获可预见异常，返回非零状态及简短错误信息，不向用户倾倒堆栈。
- 输入文件不存在、参数错误或核心数据损坏时停止受影响步骤并标记 `RUNTIME-001`；其他可读文件仍继续处理。
- DOCX结构损坏或XML不完整时，文档提取器捕获异常并返回结构化读取失败；自动建议重新导出Word、另存为可搜索PDF或提供纯文本，不把损坏文件当成无问题。
- `lxml`、`python-docx`、Pillow、本地OCR引擎或外部连接器属于按功能启用的可选能力。缺失时只停用Word原生批注、模拟评测或OCR等对应功能，不影响文本审阅和问题清单。
- 长文件包或多轮审阅使用 `scripts/review_checkpoint.py` 保存阶段状态和源文件指纹。16个内部步骤各自都是保存点；一个步骤内每完成一个文件或条款族，用 `autosave --unit <稳定编号>` 增量保存。进程中断后运行 `resume`；源文件变化时自动把依赖阶段标为需要重做，并保留上一份状态备份用于回滚。

Agent需要定位运行问题时，可使用以下最小命令；普通用户无需执行：

```text
python3 scripts/runtime_self_check.py --format json
python3 scripts/extract_contract_text.py <file> --format json
python3 scripts/review_checkpoint.py autosave <state.json> <stage> --unit <file-or-clause-id> --artifact <path>
python3 scripts/review_checkpoint.py resume <state.json> --source <files...>
python3 scripts/benchmark_lookup.py "liquidation preference" --json
python3 scripts/validate_skill_consistency.py
```

进度文件只保存文件指纹、阶段状态、已完成单元的稳定编号和产物路径，不保存合同正文、批注内容或审阅结论。回滚仅恢复最近一次写入前的 `.bak`，不会修改用户原始交易文件。

## 高频问答

### 文件不齐，是否必须等齐再开始？

不必。可以先审现有文件，但必须列出缺失文件及受影响的跨文件判断，补齐后再更新结果。

### 扫描件OCR出了一部分文字，是否算完成？

不算全文完成。必须列出失败页，并仅对确认读取到的部分作有限审阅。

### 没有上一版，能否判断对方接受了哪些意见？

不能。可以审当前文本风险，但不得把当前文本推断为对上一轮意见的接受或拒绝。

### 外部数据库不可用，会不会整项停止？

不会。继续完成文件读取、条款分析和可核验部分，并明确哪些主体事实或法律依据仍待确认。

### 谁应当确认法律事项？

涉及中国法律相关事项，请咨询执业中国律师；涉及非中国法管辖事项，请咨询相应法域的执业律师。
